# 🏆 Case Study: Full-Cycle E-commerce Analytics Infrastructure (Olist Dataset)

## 📌 Situation (Ситуація)

Компанія (великий бразильський e-commerce маркетплейс Olist) накопичила величезний масив сирих, децентралізованих даних про замовлення, логістику та фінанси. Менеджмент трьох ключових департаментів (**Маркетинг, Логістика, Фінанси/СЕО**) приймав рішення наосліп або на основі застарілих звітів через відсутність єдиної аналітичної екосистеми. Головною технічною проблемою була наявність критичної кількості дублікатів транзакцій у сирих логах та затирання або некоректний формат дат доставки, що викривлювало фінансову звітність та метрики ефективності операторів.

## 🎯 Task (Завдання)

Необхідно було з нуля спроектувати та впровадити наскрізну аналітичну інфраструктуру: від очищення та трансформації даних на рівні СУБД (PostgreSQL) до створення інтерактивної BI-звітності (Power BI).

**Ключові виклики:**

1. Очистити базу від дублікатів на рівні бекенду без втрати первинних бізнес-метрик.
2. Розрахувати точний час проходження замовлення по всій логістичній воронці.
3. Створити три окремі інтерактивні дашборди у єдиному UX/UI стилі з колірним кодуванням під бізнес-цілі кожного департаменту.
4. Оптимізувати швидкість роботи моделей — оновлення візуалізацій не повинно перевищувати 3–4 секунди.

## 🛠️ Action (Дії)

Замість хаотичного написання коду проєкт було реалізовано за методологією **Reverse Planning** (від фінальних бізнес-вимог та метрик СЕО до архітектури бази даних). Весь цикл розробки зайняв **12 годин 30 хвилин**, де 50% часу було інвестовано в problem-solving та проектування архітектури.

1. **Backend & Data Engineering (PostgreSQL):**
* Спроектував та розгорнув 3 оптимізовані бази поглядів (`VIEW`) під кожну бізнес-вертикаль.
* Для очищення транзакцій від дублікатів застосував аналітичні віконні функції (`ROW_NUMBER() OVER (PARTITION BY... )`) замість важких `DISTINCT`, що зберегло цілісність даних і прискорило запити.
* Вирішив проблему системних збоїв у датах доставки, інтегрувавши перевірку логіки статусів (`WHEN order_status = 'shipped' THEN NULL`) та валідацію за довжиною рядка (`LENGTH`), ліквідувавши аномалії розрахунку інтервалів (дикі мінуси типу $-737$ тис. днів).
* Виконав декомпозицію логістичних кроків через конвертацію Unix-epoch частот в Unix-timestamp для точного розрахунку днів.


2. **Frontend & BI Architecture (Power BI):**
* Створив єдину дизайн-систему (шапка фільтрів, адаптивна кнопка `Reset`, закруглені KPI-картки) з логічним розподілом колірних зон (Маркетинг — блакитний, Логістика — жовтий, Фінанси — глибокий зелений).
* Написав DAX-міри за допомогою конструкцій `VAR ... RETURN` (аналог `WITH` в SQL), що дозволило кешувати проміжні обчислення в оперативній пам'яті та знизити навантаження на модель.



## 📊 Result (Результат)

За **12.5 годин** розроблено та запущено в експлуатацію повноцінну BI-екосистему з трьох дашбордів із точністю розрахунків **99%+** (допустима похибка < 1%) та швидкістю відклику візуалів **< 4 секунд**:

* **Marketing Dashboard:** Підсвітив критично низький рівень клієнтського Retention (**94,39%** покупців роблять лише 1 замовлення), сформувавши ТЗ для запуску CRM-маркетингу. Зафіксував історичний аномальний пік Чорної П'ятниці 2017 року (1214 замовлень на день).
* **Logistic Dashboard (OTIF):** Розрахував загальний рівень своєчасної доставки компанії (**91,78%**). Виявив головне "вузьке горло" процесів: склад збирає товар швидко (`3,01` дні), а транспортна служба (`Carrier|Customer`) витрачає в середньому `9,19` днів. Локалізував проблемні зони на карті — штати `AL` та `MA` з рівнем запізнень >21-24% та Lead Time у ~24 дні.
* **Finance/CEO Dashboard:** Оцифрував ключові показники бізнесу: **GMV $17,2 млн**, 106,88 тис. унікальних продажів та базу в 2,923 тис. мерчантів із вбудованими Sparklines-трендами зростання. Візуалізував 100% структуру PnL витрат на логістику до виручки за місяцями, підсвітивши аномальне зростання витрат на доставку в грудні 2016 року (**44,44%** від обороту).
